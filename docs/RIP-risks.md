## 1. Entity Resolution — "ООО Ромашка" = "LLC Romashka"

**Критичность: 🔴 Высокая**

### Суть проблемы
Один и тот же человек или компания появляется из разных источников под разными именами. Без решения этой проблемы граф наполняется дублями и становится мусором.

### Инструменты
- **Splink** — probabilistic record linkage, работает без обучающих данных, поддерживает fuzzy matching
- **Dedupe** — active learning подход: задаёт вопросы по парам записей, обучается на ответах человека

### Реальная картина по типам сущностей

**Компании — решаемо:**
ИНН — однозначный идентификатор. Если ИНН есть — entity resolution тривиален.
Если ИНН нет — fuzzy matching по названию + адрес + телефон.
Инструменты справляются.

**Люди — сложно:**
"Иванов Иван" без email, телефона, должности — тупик для любого алгоритма.
Нужна комбинация: ФИО + компания + должность.
Финальное решение — всегда за менеджером.

### Решение
- У компаний: ИНН как обязательный anchor при наличии
- У людей: canonical_id + список aliases в модели данных
- Merge — только через подтверждение менеджера
- Система предлагает потенциальные дубли, не мерджит автоматически

### Открытый вопрос — Merge Logic
Когда менеджер говорит "это один человек" — что происходит?
- Кто становится canonical node?
- Как мерджатся связи от обоих?
- Что делать с дублирующимися связями?

**Это нужно прописать как бизнес-правило ДО начала разработки.**

## 2. Первичное наполнение БД — холодный старт

**Критичность: 🔴 Высокая**

### Суть проблемы
Система без данных бесполезна. Данные без участия людей не появятся.
Но заставить менеджеров верифицировать тысячи записей — нереально.

### Реальная оценка источников

**Outlook-контакты:**
- Плоские данные: имя, email, телефон, иногда компания и должность
- Связей между людьми нет вообще
- Импорт автоматический, но граф из этого не построишь
- Ценность: seed нод, не seed связей

**Старые PDF-отчёты:**
- Там реальные связи — самый ценный источник
- AI extraction даёт ~60–70% точности
- Остальное требует верификации
- На 10 проектов × 20 человек = ~200 верификаций — терпимо
- На 50 проектов — уже организационная проблема

### Главный риск
Менеджеры сделают первичный онбординг и бросят верификацию на полпути.
Граф остаётся полупустым → система не даёт ценности → заказчик разочарован.
**Это убивает проект не технически, а организационно.**

### Решение
Первичный онбординг — наша работа, не клиента:
1. Берём 5–10 лучших закрытых проектов заказчика
2. Менеджер вносит данные вручную через AdminJS (extraction не используется)
3. DaData автоматически обогащает все найденные компании по ИНН
4. Сдаём заказчику уже заполненную базу

Это часть стоимости PoC, явно прописать в договоре.

## 3. Версионирование данных

**Критичность: 🟡 Средняя**

### Суть проблемы
Данные меняются: люди меняют работу, компании реорганизуются, связи рвутся.
Нужно хранить историю, не перезаписывать.

### Два подхода

**Temporal tables (рекомендуется для MVP):**
- У каждой записи: `valid_from` / `valid_to`
- Текущая версия: `valid_to IS NULL`
- PostgreSQL поддерживает нативно
- Просто реализовать, просто запрашивать

**Event sourcing (избыточно для MVP):**
- Хранишь не состояние а события: "связь добавлена", "нода изменена"
- Полный audit trail
- Сложнее, дороже, нужно позже

### Реальная сложность — не хранение, а merge
Temporal tables решают хранение. Но merge двух нод — отдельная задача:
- Что происходит с историей обеих нод после слияния?
- Как показывать историю merged node?
- Можно ли отменить merge?

Это нужно решить на уровне модели данных до разработки.

## 4. Human-in-the-loop — объём верификации

**Критичность: 🟡 Средняя**

### Три точки где нужна верификация

**При extraction (разовая работа):**
AI прочитал отчёт → предлагает список сущностей → менеджер смотрит и правит.
Объём конечный, терпимо.

**При merge (очередь задач):**
Система нашла потенциальные дубли → спрашивает "это один человек?".
Нужен отдельный UI — очередь задач на верификацию.
Без этого UI merge не работает на практике.

**При обогащении (постоянный поток):**
AI нашёл новую информацию из внешних источников → предлагает добавить.
Менеджер подтверждает или отклоняет.
Это ongoing процесс — нужна приоритизация.

### Главный риск
Если верификаций накапливается слишком много — менеджер их игнорирует.
Очередь растёт → данные не обновляются → система деградирует.

### Решение
- Показывать только высококритичные решения (ЛПР/ЛВР, merge)
- Остальное применять автоматически с низким confidence
- Жёсткий лимит на размер очереди верификации: не больше N задач одновременно
- Геймификация / нотификации — опционально в v2

## 5. Холодный старт — что система показывает когда граф пустой

**Критичность: 🔴 Высокая**

### Суть проблемы
После MVP деплоя система ничего не знает. Менеджер открывает новый проект — система говорит "связей не найдено". Заказчик разочарован, считает что система не работает.

### Решение
Холодный старт должен быть решён ДО сдачи MVP:
- Первичный онбординг (см. пункт 2) закрывает это
- На момент демо заказчику — база уже заполнена реальными данными
- Первый показ системы — только на реальных данных их же проектов

Никогда не показывать систему на пустой базе.

## 6. AI Extraction — качество на русскоязычных PDF

**Критичность: 🟡 Средняя**

### Суть проблемы
Большинство LLM хуже работают с русскоязычными текстами чем с английскими.
PDF-отчёты строительной фирмы — специфическая терминология, аббревиатуры, неформальный язык в комментариях менеджеров.

### Что нужно проверить до разработки
- Прогнать 2–3 реальных отчёта через выбранный LLM
- Оценить качество extraction вручную
- Если результат плохой — настроить промпт или поменять модель

**Это нужно сделать на этапе пресейла, до подписания договора.**

## 7. Adoption — менеджеры вернутся к привычным инструментам

**Критичность: 🔴 Высокая**

### Суть проблемы
Менеджеры строительной фирмы работают через телефон и личные встречи. Система требует от них новых привычек. Типичный сценарий провала: первые две недели активны, потом возвращаются к Excel и WhatsApp. Граф перестаёт обновляться, данные устаревают, система умирает.

### Решения

**Организационное — система как точка старта проекта:**
Договорённость с заказчиком: новый проект начинается с создания карточки в системе. Не параллельно — до. Это не технология, это регламент. Но он работает.

**Техническое — push вместо pull:**
Refresh Worker + Telegram уведомления. Менеджер получает сигнал в Telegram → открывает систему по конкретному поводу. Система приходит к нему, не он к системе.

**Ключевой аргумент для заказчика:**
Один найденный вход в проект = потенциальный контракт на десятки миллионов. ROI системы считается за один случай. Это мотивирует менеджеров пользоваться системой даже если это требует усилий.

## 8. Connector Pattern — риск неправильной реализации

**Критичность: 🟡 Средняя**

### Суть проблемы
Connector Pattern — главная долгосрочная ценность системы. В v2 и v3 система обогащается автоматически через новые коннекторы (тендеры, новости, соцсети). Если в MVP коннектор DaData реализован как хардкод а не через интерфейс — добавление каждого нового источника будет дорогим рефакторингом.

### Решение
С первого дня: Source Connector как явный интерфейс.
```
interface SourceConnector {
  fetch(entityId, params): Observation[]
  capabilities(): string[]
  rateLimit(): RateLimitConfig
}
```
DaData — первая реализация этого интерфейса. Все остальные коннекторы в v2/v3 — новые реализации без изменения бизнес-логики.

**Это нужно заложить в архитектуру на старте, не добавить потом.**

## 9. Устаревание данных о физлицах

**Критичность: 🟡 Средняя**

### Суть проблемы
DaData отслеживает изменения в компаниях (смена директора, статус). Но физлица — не отслеживаются автоматически. Иванов уволился из компании год назад, но в графе всё ещё числится сотрудником. Менеджер идёт через него как через "внутренний контакт" — а он уже не там.

### Решение
Пока не решено. Варианты:
- Менеджер вручную помечает связь как устаревшую
- Система периодически спрашивает "эта связь ещё актуальна?" для старых edges
- confidence связи автоматически снижается со временем (decay)

**Открытый вопрос — выбрать подход до разработки.**

## 10. UI для конфликта данных

**Критичность: 🟡 Средняя**

### Суть проблемы
DaData говорит директор Иванов. Менеджер говорит директор Петров. Два источника дают разные данные об одном факте. Confidence решает приоритет — но нужен UI где менеджер видит конфликт и принимает решение.

### Решение
Отдельная очередь конфликтов в AdminJS — аналогично очереди merge. Показывает: факт А (источник, confidence) vs факт Б (источник, confidence). Менеджер выбирает правильный или помечает оба как сомнительные.

## 13. Двунаправленный граф — реализация не прописана

**Критичность: 🔴 Высокая**

### Суть проблемы
Решено что граф двунаправленный — "А знает Б" = "Б знает А". Но в схеме БД `relationships` хранит `from_id/to_id` как направленные записи. Нигде не прописано как именно это реализуется.

### Решение — нужно выбрать до разработки
**Вариант А:** хранить одну запись, при запросах искать `WHERE from_id = X OR to_id = X`. Проще в хранении, чуть сложнее в запросах. **Рекомендуется.**

**Вариант Б:** хранить две записи (A→B и B→A) с одним `relationship_id`. Проще запросы, дублирование данных.

## 14. NER для Pseudonymization — библиотека не выбрана

**Критичность: 🔴 Высокая**

### Суть проблемы
Pseudonymization требует NER pre-pass — найти все имена, телефоны, email в тексте перед заменой на токены. Для русскоязычных текстов нетривиально. Библиотека не выбрана.

### Варианты
- **natasha** — лёгкая русскоязычная NER, хорошо работает с именами
- **spaCy + ru_core_news** — мощнее, но тяжелее
- **LLM сам находит PII** — первый проход LLM только для нахождения PII

**Нужно:** прогнать 2-3 реальных текста через natasha и spaCy, сравнить качество, выбрать до разработки Extraction Service.

## 15. assigned_manager_id — отсутствует в схеме БД

**Критичность: 🟡 Средняя**

### Суть проблемы
Refresh Worker должен уведомить менеджера через Telegram. Но в схеме нет `assigned_manager_id` на проекте и нет таблицы `users`.

### Решение
Добавить таблицу `users`:
```sql
id               UUID PRIMARY KEY
name             TEXT NOT NULL
telegram_chat_id TEXT
role             TEXT  -- manager | admin
created_at       TIMESTAMPTZ DEFAULT now()
```

Добавить на `projects`:
```sql
assigned_manager_id  UUID REFERENCES users(id)
```

3. Оценка риска (низкий/средний/высокий)

## 17. Версионирование — согласованность исторических запросов

**Критичность: 🟡 Средняя (MVP), 🔴 Высокая (v1+)**

### Суть проблемы
Temporal tables выглядят просто — `valid_to IS NULL` для текущего состояния. Но исторические запросы ("покажи граф на дату X") требуют что ВСЕ части запроса фильтруются по одной дате. Если в одном join забыл фильтр по дате — граф химерный: часть нод из прошлого, часть из настоящего.

Дополнительные сложности:
- Merge с историей — у двух Person своя история версий, как показывать историю merged ноды?
- Cascade версионирование — если компания изменила статус, нужно ли версионировать связанные факты?

### Решение для MVP
Только текущее состояние через `valid_to IS NULL`. Исторические запросы — v1+. До их реализации — явный запрет в коде на запросы без фильтра по дате.

## 18. Confidence агрегация из нескольких источников — алгоритм не определён

**Критичность: 🔴 Высокая**

### Суть проблемы
Один и тот же факт подтверждается из трёх источников. Как агрегируется confidence? Нигде не описано.

Варианты:
- **Максимум** — берём наивысший confidence. Просто, но игнорирует подтверждение.
- **Среднее** — усредняем. Не учитывает что manual важнее AI.
- **Байесовское обновление** — каждый новый источник обновляет вероятность. Правильно, но сложно.
- **Взвешенное максимальное** — `max(confidences) × (1 + 0.1 × count_sources)`. Бонус за подтверждение, ограниченный рост.

### Рекомендация
Взвешенное максимальное для MVP — просто реализовать, учитывает подтверждение:
```
final_confidence = min(0.99, max_confidence × (1 + 0.05 × (source_count - 1)))
```

**Нужно решить до разработки Fact creation логики.**

## 19. Confidence при конфликте источников — поведение не описано

**Критичность: 🟡 Средняя**

### Суть проблемы
DaData говорит директор Иванов, менеджер говорит Петров. Что показывает система до того как менеджер разрешил конфликт в AdminJS?

### Решение
Показывать fact с наивысшим confidence как "текущий", остальные конфликтующие — с тегом `conflicted = true`. В UI показывать предупреждение "⚠ Есть конфликтующие данные" на карточке.

## 20. Feedback → weight_config — петля не замкнута

**Критичность: 🟡 Средняя (MVP), 🔴 Высокая (v1+)**

### Суть проблемы
Менеджер ставит 👎 на факт — сигнал что источник дал неверные данные. В MVP это только логируется. Но кто и когда обновляет `weight_config`? Без ответа веса никогда не улучшаются.

### Решение
**MVP:** только логирование. Администратор вручную просматривает `feedback_events` раз в месяц и корректирует `weight_config`.

**V1:** автоматический пересчёт. Если для `(source, fact_type)` накопилось >10 негативных отзывов — система предлагает понизить confidence. Финальное решение за admin.
