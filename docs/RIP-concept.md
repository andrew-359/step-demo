# Relationship Intelligence Platform (RIP)
## Документация проекта — живой документ

> Статус: черновик v0.1 | Последнее обновление: начало работы

---

## 1. Контекст и проблема

**Заказчик:** строительная фирма, B2B, мало проектов но высокая стоимость каждого.

**Проблема:** менеджеры вручную собирают данные о людях и компаниях под каждый проект. Знания о связях между людьми — в головах менеджеров и разрозненных PDF-отчётах. При смене менеджера или начале нового проекта всё начинается с нуля.

**Ключевая бизнес-задача:** как можно быстрее найти точку входа в проект — конкретного человека, через которого можно выйти на ЛПР/ЛВР.

---

## 2. Главная идея

Система — это **корпоративная память о связях**, оформленная в граф.

```
Не CRM (не pipeline сделок)
Не база контактов (не адресная книга)
→ Граф рукопожатий: кто с кем связан, через кого заходить
```

Каждый проект — это подграф внутри мастер-графа.
Мастер-граф накапливается со временем и не удаляется.

---

## 3. Главный сценарий (MVP)

```
1. Появился новый проект
2. Менеджер добавляет компанию / известных участников
3. Система ищет пересечения с мастер-графом
4. Показывает: через кого можно зайти, кто ЛПР/ЛВР
5. Менеджер верифицирует, добавляет комментарии
6. Граф обновляется — знания остаются в системе
```

## 4. Сущности

### Основные ноды графа

| Сущность | Описание |
|----------|----------|
| Person | Физическое лицо |
| Company | Юридическое лицо / организация |
| Project | Конкретный проект (внутренний или внешний) |

### Типы связей (edges)

| Связь | Описание |
|-------|----------|
| works_at | Person → Company (должность, период) |
| knows | Person ↔ Person (тип знакомства, источник) |
| participates_in | Person/Company → Project (роль) |
| affiliated_with | Company ↔ Company (структура, партнёрство) |

### Метаданные каждой связи

- `strength` — сила связи (0–1)
- `confidence` — достоверность (0–1)
- `source` — откуда данные (manual / report / parsing / ai)
- `valid_from` / `valid_to` — временной период
- `version` — история изменений

### Теги на нодах Person

- `is_lpr` — лицо принимающее решение
- `is_lvr` — лицо влияющее на решение
- `notes` — свободный текст менеджера (самое ценное)

## 5. Принципы работы с данными

**Ничего не удаляется** — только версионируется.

**Данные всегда грязные** — это норма. Система хранит вероятности, не ищет абсолютную истину.

**Вероятностная модель — не баг, а фича.**
Система не говорит "директор компании — Иванов". Она говорит "с вероятностью 85% директор — Иванов, источник ЕГРЮЛ". Это честнее и полезнее чем ложная уверенность. Менеджер видит confidence и сам решает насколько доверять данным.

**Human-in-the-loop обязателен** — AI предлагает, менеджер подтверждает. Особенно для ЛПР/ЛВР.

**Confidence по источнику:**
```
manual (менеджер)     → 0.9–1.0
DaData / реестры      → 0.4–0.9  (зависит от типа факта, см. weight_config)
старый отчёт (AI)     → 0.6–0.8
парсинг               → 0.4–0.6
AI enrichment         → 0.3–0.5
```

## 6. Как наполняется граф (источники данных)

### Старт (ретроспективный онбординг) — делают разработчики
- Загрузка старых PDF-отчётов по закрытым проектам (5–10 проектов)
- AI извлекает сущности и связи
- Менеджер верифицирует результат через AdminJS
- DaData автоматически обогащает все найденные компании по ИНН
- Система сдаётся заказчику уже с заполненным графом

### MVP — текущая работа
- **DaData API** — основной внешний коннектор. До 10 000 запросов/день бесплатно. Даёт: название компании во всех вариантах, руководители, учредители, адрес, статус. ИНН как anchor для entity resolution.
- Ручной ввод менеджером через чатбот (7 команд)
- Менеджерские комментарии к нодам
- Refresh Worker — ночной cron, отслеживает изменения по активным проектам через DaData, уведомляет в Telegram

**Outlook:** хранит телефоны и имена контактов. Импорт через CSV экспорт — разовый скрипт при онбординге. Даёт seed нод (Person + телефон). Связей между людьми нет — только ноды. Парсинг переписки не планируется.

### V2+
- Парсинг новостей и публичных источников
- Тендеры и госзакупки
- Социальные сети / Telegram
- Контур / СПАРК (платные API для глубокого анализа компаний)

### Connector Pattern — главный архитектурный принцип
Каждый источник подключается через единый интерфейс Source Connector.
Добавление нового источника в v2 = новая реализация интерфейса, без изменения бизнес-логики.
**Это главная долгосрочная ценность системы** — граф обогащается автоматически по мере подключения новых коннекторов.

## 7. Безопасность данных

### Pseudonymization pipeline

```
Реальные данные
      ↓
[Pseudonymization layer]
  Person → PERSON_001
  Company → COMPANY_047
  Phone → PHONE_003
      ↓
LLM (видит только токены)
      ↓
Ответ с токенами
      ↓
[Обратная замена]
      ↓
Реальные данные в отчёте
```

**Маппинг-таблица** хранится только на сервере заказчика. Никогда не передаётся наружу.

**Ограничение:** enrichment через внешние знания LLM о конкретных людях — невозможен при анонимизации. Это осознанный trade-off в пользу безопасности.

## 8. LLM — connector pattern

Абстракция провайдера: одна система, любой LLM за интерфейсом.

```
Business logic
      ↓
[LLM Connector Interface]
      ↓
┌─────────────┬──────────────┬────────────┐
Self-hosted    GigaChat/      OpenAI/
(Ollama)       YandexGPT      Anthropic
```

**Приоритет для MVP:** Self-hosted (Ollama + Llama/Mistral) или GigaChat.
- Self-hosted: данные не покидают периметр, нет зависимости от интернета
- GigaChat/YandexGPT: данные в РФ, стабильный API, юридически чисто

Зарубежные провайдеры — опционально через connector, не в основном пути.

## 10. Graph Layer

**Решение по БД:** PostgreSQL + recursive CTE.
На масштабе 5к контактов / 100 проектов — достаточно.

**Критерий перехода на Neo4j:** если graph queries начинают занимать более 500ms на типичном запросе shortest_path. Не раньше.

**Основные операции:**
- shortest path (цепочка рукопожатий между двумя людьми)
- **weighted path** (кратчайший путь с учётом качества связей — не только длина)
- centrality (кто ключевой узел в проекте)
- subgraph extraction (подграф по проекту)
- multi-hop traversal (найди всех на расстоянии N от Person X)

**Принципы графа:**
- **Двунаправленный** — "Иванов знает Петрова" = "Петров знает Иванова". Edges хранятся один раз, поиск идёт в обе стороны.
- **Контакты расшарены** — все менеджеры компании видят все контакты. Приватности между менеджерами нет.
- **Старт поиска пути** — от всех менеджеров компании одновременно (multi-source shortest path). Не от одного конкретного менеджера.
Длина пути ≠ качество пути. Три рукопожатия через друзей и партнёров могут быть лучше чем два через формальную иерархию. Сотрудник не будет продвигать тебя к своему директору — это риск для его карьеры.

Вес связи считается из комбинации:
```
weight = relationship_type_score × strength × direction_penalty × freshness_score

relationship_type_score:
  friends        → 1.0
  ex_colleagues  → 0.8
  partners       → 0.7
  colleagues     → 0.4   ← горизонтальные связи внутри иерархии рискованны
  works_at       → 0.2   ← формальная иерархия — самый рискованный путь

direction_penalty:
  горизонтальные связи → 1.0
  вертикальные вверх   → 0.5  ← сотрудник к директору
  вертикальные вниз    → 0.8  ← директор к сотруднику

freshness_score (на основе last_interaction_at):
  < 3 месяцев   → 1.0   ← тёплая связь
  3–12 месяцев  → 0.8
  1–2 года      → 0.5
  > 2 лет       → 0.3   ← холодная связь, требует "разогрева"
  NULL          → 0.6   ← неизвестно, нейтральный вес
```

Система показывает **оба варианта**: оптимальный по длине и оптимальный по качеству. Менеджер сам выбирает стратегию.

## 11. Стратегический смысл модели данных

Все метаданные которые мы закладываем в граф — веса связей, типы отношений, confidence, направленность, роли ЛПР/ЛВР, временные метки — это не просто атрибуты для хранения. Это **заготовка под AI анализ**.

**Семантика связей — главное отличие от адресной книги:**
Граф бесполезен если хранит только факт "А знает Б". Ценность появляется когда граф знает:
- "знает" vs "может помочь" — разные вещи
- "работал с" vs "есть влияние" — разные вещи
- "через кого заходить" зависит от типа задачи и контекста человека

Для этого в каждой связи хранится `context` (свободный текст менеджера) и структурированные атрибуты в `entity_attributes`.

**Живая система — принцип непрерывного улучшения:**
```
Новый проект
  → менеджер вводит данные и notes
  → AI извлекает структурированные атрибуты из notes
  → weighted path учитывает атрибуты при поиске пути
  → менеджер утверждает или отклоняет стратегию
  → feedback обновляет веса атрибутов
  → следующий проект — система знает больше и рекомендует точнее
```

**Гибкое хранилище мета-атрибутов (`entity_attributes`):**
Любой новый признак который окажется важным для AI — добавляется без изменения схемы БД. Сегодня это `approach_via` и `contact_preference`. Завтра система сама обнаружит что какой-то признак коррелирует с успешными входами — и он станет частью модели.

**Raw data как долгосрочный актив:**
Observations хранятся всегда — это не просто кэш, это датасет. Появится более умный AI через год — прогоняем те же сырые данные заново без повторного скрапинга. Инвестиция в данные не обесценивается.

**Ensemble подход (v3):**
Один и тот же граф можно прогнать через несколько LLM параллельно и сравнить результаты. LLM Connector как абстракция уже позволяет это без изменения кода.

**Граф — единственный источник истины:**
Не LLM, не отчёты — граф. LLM это инструмент интерпретации данных в графе. Граф живёт дольше любой модели.

**Самая сложная техническая задача.**

Проблема: один человек может появиться как "Иванов И.И.", "Ivan Ivanov", `ivanov@company.ru` в разных источниках.

Подход:
1. Fuzzy matching по имени + должности + компании
2. AI для неочевидных случаев
3. Менеджер финально подтверждает merge
4. Merge не удаляет — создаёт canonical node со ссылками на источники

**Важно:** заложить в модель данных с самого начала. Переделывать потом — дорого.

## 14. Роадмап

**PoC (2–3 недели)**
Ручной ввод 2–3 реальных проектов + Outlook CSV импорт. Граф. Один вопрос: "через кого зайти". Проверка гипотезы на реальных данных.

**MVP (после PoC) — рабочий инструмент**
Чатбот 8 команд + AdminJS верификация + DaData коннектор + Refresh Worker + Telegram уведомления + Keycloak + Pseudonymization + LLM Connector + базовый Project Report (участники, ЛПР/ЛВР, PDF).

**V1 (после MVP) — умная аналитика**
Стратегии входа с AI reasoning + Relationship Map в отчёте + Feedback система + Conflict resolution UI + weighted path визуализация.

**V2 (после V1) — автоматизация**
Новые коннекторы (тендеры, новости) + интерактивная визуализация графа + Rank Worker + Strategy Engine + Redis/BullMQ + векторный поиск.
