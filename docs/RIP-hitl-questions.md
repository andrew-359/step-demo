## Контекст 1 — Онбординг
*Разовый, разработчик вносит, менеджер валидирует*

Вопросов нет — решение принято:
- Разработчики вносят данные вручную
- Менеджер верифицирует через AdminJS очередь
- Происходит один раз при запуске

## Контекст 2 — Merge дублей
*Редкий, но критичный — неправильный merge ломает граф*

- [ ] **Кто имеет право делать merge — любой менеджер или только admin?**
  Неправильный merge сложно откатить. Возможно нужна роль "senior manager" или только admin.

- [ ] **Можно ли отменить merge после подтверждения?**
  Если да — как выглядит UI и что происходит со связями которые были созданы после merge?

- [ ] **Как выглядит UI для merge?**
  Две карточки рядом, кнопка "это один человек". Какие данные показывать чтобы решение было осознанным?

## Контекст 3 — Сигналы от Refresh Worker
*Регулярный — система нашла изменение, менеджер получил уведомление в Telegram*

- [ ] **По умолчанию принимаем или ждём подтверждения?**

  Вариант А — автоматически применяем с низким confidence, менеджер может оспорить.
  Плюс: граф всегда актуален. Минус: может засориться мусором.

  Вариант Б — данные висят как "pending", ждём подтверждения менеджера.
  Плюс: граф чистый. Минус: требует действия на каждый сигнал, менеджер может игнорировать.

- [ ] **Как менеджер отвечает на сигнал из Telegram?**
  Кнопки прямо в сообщении (✅ Принять / ❌ Отклонить) или нужно заходить в систему?

- [ ] **Что если менеджер не отреагировал на сигнал N дней?**
  Автоматически применить? Напомнить? Эскалировать к admin?

## Контекст 4 — Оценка отчёта и данных
*Постоянный — менеджер работает с отчётом и оценивает качество данных*

### Оценка фактов и сигналов (👍 / 👎 / ⚠)

- [ ] **Оценка факта — это просто запись или сразу меняет граф?**

  Вариант А — логируется в feedback_events, применяется batch в v2.
  Вариант Б — сразу меняет confidence факта/связи.

  Что быстрее реализовать в MVP? Что правильнее архитектурно?

- [ ] **Кто видит оценки других менеджеров?**
  Если два менеджера работают с одной компанией и один поставил ⚠ — второй это видит?
  Или оценки приватные?

### Утверждение отчёта (draft → approved)

- [ ] **Утверждение — обязательный шаг или опциональный?**
  Менеджер обязан утвердить перед тем как взять отчёт на встречу — или это просто статус для порядка?

- [ ] **Что происходит с отчётом после утверждения?**
  Замораживается (нельзя изменить)? Или можно обновить и утвердить снова?

- [ ] **Нужна ли история версий отчёта?**
  Утвердили v1, потом обновили данные, система предлагает пересоздать — старый отчёт сохраняется?

## Общие вопросы по всем контекстам

- [ ] **Сколько менеджеров будет работать с системой одновременно?**
  Влияет на дизайн очередей верификации и на видимость оценок.

- [ ] **Нужен ли audit log — кто что подтвердил и когда?**
  Для enterprise это обычно обязательно. Для этого заказчика — вопрос.

- [ ] **Как приоритизировать очередь верификации?**
  По дате? По важности проекта? По типу сигнала?
  Без приоритизации менеджер видит 50 задач и игнорирует все.
