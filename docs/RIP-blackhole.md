## Черновик гипотез до финализации видения

> Статус: рабочая свалка смысла, не ТЗ.
> Назначение: фиксировать спорные архитектурные и продуктовые идеи до того,
> как они будут перенесены в `RIP-concept.md`, `RIP-mvp-roadmap.md` или `RIP-tech-spec.md`.

## 1. Новый базовый тезис

Текущие документы описывают RIP как граф связей и админку вокруг отчета.
Но более точная рамка может быть шире:

```
RIP = интерактивная база строительной проектной памяти
      + граф связей
      + AI/HITL-оператор
      + плагины источников и рабочих контуров
```

Главная ценность не в том, что есть “админка”.
Главная ценность в том, что система становится живым рабочим контуром,
где человек и AI вместе собирают, проверяют и используют знания о проектах,
людях, компаниях, объектах и точках входа.

## 2. Core vs Plugins

### Core

Core должен отвечать за то, без чего продукт теряет смысл.

Предварительно в core попадают:

| Компонент | Почему core |
|---|---|
| База данных | Главный источник правды, память системы |
| Граф сущностей и связей | Основная модель поиска точки входа |
| HITL-очередь решений | Без человека система быстро загрязняет граф |
| Entity resolution / merge | Качество графа зависит от борьбы с дублями |
| Observation → Signal → Fact → Snapshot | Базовый pipeline доверия к данным |
| История / версионирование | Строительные проекты длинные, данные меняются |
| Права и audit log | Кто подтвердил факт, merge или отчет |
| AI orchestration | AI должен уметь вызывать допустимые действия системы |

### RAG: core или plugin

Открытый вопрос.

Аргумент за core:
- AI-агенту нужен общий слой памяти и поиска по документам.
- Отчеты, переписка, PDF, регламенты и протоколы становятся частью проектной памяти.
- Если RAG вынести слишком далеко, AI будет слабее связан с фактами графа.

Аргумент за plugin:
- RAG может иметь разные реализации: локальный, облачный, per-project, per-source.
- Не все клиенты захотят одинаковый режим хранения документов.
- Для MVP можно начать с простого document search вокруг core.

Промежуточная гипотеза:

```
Core хранит документы, источники, чанки, ссылки на факты и права доступа.
RAG engine / vector search provider может быть plugin.
```

### CRM: plugin

CRM-контур почти точно должен быть plugin, а не core.

Причины:
- RIP не должен превращаться в сделочную CRM.
- У клиента уже может быть Bitrix24, amoCRM, Excel или своя дисциплина.
- CRM нужна как рабочий контур вокруг графа: задачи, встречи, follow-up, статусы контактов.
- Core должен знать, что контакт/встреча/задача существует, но не обязан быть CRM-системой.

Гипотеза:

```
Core: Person, Company, Project, Relationship, Fact, Signal, Strategy.
CRM plugin: Meeting, Task, Follow-up, Deal-ish status, calendar sync, reminders.
```

## 3. Что значит “интерактивная БД”

База не должна быть пассивной таблицей.
Она должна быть операционной средой, где пользователь и AI могут:

- создавать и связывать сущности;
- проверять факты;
- задавать вопросы к графу;
- видеть, какие данные подтверждены, спорны или устарели;
- запускать сценарии: “найди вход”, “проверь компанию”, “подготовь встречу”;
- получать предложения от AI;
- подтверждать или отклонять изменения;
- видеть последствия решения до применения.

Рабочая формула:

```
Interactive DB = data model + graph + actions + HITL + audit + AI-readable UI flows
```

## 4. AI-агент и голос

Целевое ощущение продукта:

```
Пользователь говорит голосом:
"Создай проект по новому объекту в Шушарах,
добавь Company Name, проверь ЛПР и покажи через кого можно зайти".

AI:
1. распознает намерение;
2. показывает план действий;
3. вызывает разрешенные flow actions;
4. просит подтверждение на опасных шагах;
5. обновляет граф и протокол;
6. объясняет, почему предлагает конкретную стратегию.
```

Важно: AI не должен быть свободным “магическим чатом”.
Он должен работать как оператор над декларативными сценариями.

## 5. UI через flow configs

Гипотеза: интерфейс нужно описывать не только компонентами, а flow-конфигами.

Flow config описывает:

- цель сценария;
- какие сущности участвуют;
- какие поля нужны;
- какие действия доступны;
- какие действия требуют HITL;
- какие статусы возможны;
- какие AI skills/tools можно вызвать;
- какие UI-представления показывать на каждом шаге.

Пример направления:

```ts
const createProjectFlow = {
  id: 'project.create_from_company',
  goal: 'Создать проект и найти точку входа',
  entities: ['Project', 'Company', 'Person', 'Relationship'],
  steps: [
    {
      id: 'company_input',
      view: 'company-search',
      action: 'source.dadata.lookupCompany',
      output: 'CompanyCandidate[]',
    },
    {
      id: 'company_confirm',
      view: 'hitl-confirm-candidate',
      requiresHuman: true,
      action: 'entity.company.attachToProject',
    },
    {
      id: 'entry_search',
      view: 'entry-paths',
      action: 'graph.findEntryPaths',
      output: 'EntryStrategy[]',
    },
    {
      id: 'strategy_confirm',
      view: 'strategy-selection',
      requiresHuman: true,
      action: 'snapshot.approveStrategy',
    },
  ],
}
```

Такой подход нужен, чтобы AI мог не “угадывать UI”, а работать по контракту:

```
skill/agent читает flow config → понимает доступные действия → вызывает безопасный шаг
```

## 6. HITL как продуктовый центр

HITL нужно вынести из “подтвердить поле” в отдельный уровень продукта.

Типы HITL-решений:

| Тип решения | Почему важно |
|---|---|
| Подтвердить ЛПР/ЛВР | Центральная ценность: кто принимает или влияет на решение |
| Подтвердить связь | Неверная связь ломает стратегию входа |
| Merge дублей | Неверный merge портит граф системно |
| Разрешить конфликт источников | DaData, отчет и менеджер могут спорить |
| Принять сигнал Refresh Worker | Новые данные не всегда можно применять автоматически |
| Утвердить стратегию входа | AI предлагает, человек выбирает |
| Утвердить отчет / snapshot | Нужно понимать, что можно нести директору или на встречу |

Ключевой UX-принцип:

```
Каждое HITL-решение должно показывать:
что изменится в графе,
почему система это предлагает,
какой источник,
какой confidence,
какой риск ошибки,
можно ли откатить.
```

## 7. Сущности на графиках: вопрос не закрыт

В текущих документах есть базовые ноды:

- Person;
- Company;
- Project.

Но для строительного продукта этого может быть мало.
Нужно отдельно решить, какие сущности должны быть нодами графа,
а какие должны быть атрибутами, фактами или представлениями.

Кандидаты в ноды:

| Сущность | Вопрос |
|---|---|
| Object / ConstructionObject | Объект строительства может быть важнее Project |
| Tender / Lot | Лоты могут иметь свои связи и ЛПР |
| Department | Отдел закупок, финблок, техзаказчик могут быть точками входа |
| Role | ЛПР/ЛВР/техзаказчик как роль во времени, а не свойство Person |
| Meeting | Нода или CRM-plugin entity? |
| Document | Нода графа знаний или только источник фактов? |
| Signal | Нода для аналитики или событие pipeline? |
| Strategy | Нода/артефакт, который утверждает менеджер |

Открытый вопрос:

```
Что именно директор должен видеть на графике:
людей и компании,
объекты и лоты,
цепочки входа,
роли влияния,
источники/факты,
или все это разными слоями?
```

## 9. Что это меняет для текущей презентации

Текущая презентация должна показывать не “админку с кучей карточек”, а:

1. Живую базу проектной памяти.
2. Граф как способ найти точку входа.
3. AI-оператора, который помогает работать с базой.
4. HITL как контроль качества и доверия.
5. Плагины вокруг core: CRM, источники, отчеты, уведомления.

Для директора главный сценарий:

```
Голосом или текстом:
"Найди вход в новый проект".

Система:
1. создает/открывает проект;
2. подтягивает компанию и объект;
3. показывает найденных людей, роли, источники;
4. просит подтвердить критичные факты;
5. строит цепочки входа;
6. предлагает стратегии;
7. фиксирует выбранную стратегию и следующие действия.
```

## 11. Рабочая формулировка для команды

Мы строим не CRM и не отчетогенератор.

Мы строим интерактивную базу проектной памяти для строительной компании,
где граф показывает точки входа в проекты,
AI помогает работать с данными голосом и текстом,
а человек подтверждает все решения, которые могут изменить доверие к графу.

Core — данные, граф, HITL, audit и AI-оркестрация.
CRM, источники, отчеты, уведомления и провайдеры поиска — плагины вокруг core.

## 12. AI-assisted forms: AI заполняет, человек утверждает

Важная продуктовая граница:

```
AI не должен сам коммитить критичные изменения в базу.
AI должен открыть нужную форму, заполнить черновик и объяснить источники.
Человек проверяет и сам нажимает финальную кнопку.
```

Это может стать базовым UX-паттерном всей системы.

### Почему это важно

- Менеджер не заполняет все руками.
- AI не ломает граф без контроля.
- У каждого поля есть источник и уровень доверия.
- Человек остается владельцем решения.
- Audit log остается понятным: кто именно утвердил изменение.

### Пример пользовательского сценария

Пользователь говорит голосом:

```
"Добавь Петрова.
Он гендир Company Name, возможно ЛПР.
Источник — Сергей сказал на созвоне."
```

Система:

1. распознает намерение `add_person`;
2. открывает форму `Person`;
3. заполняет черновик;
4. подсвечивает поля, заполненные AI;
5. показывает источник и confidence;
6. показывает, какие связи будут созданы;
7. ждет решения человека.

Человек выбирает:

- `Сохранить черновик`;
- `Сохранить и добавить связь`;
- `Подтвердить как ЛПР`;
- `Отклонить`;
- `Попросить уточнить`.

## 13. Form configs как контракт между UI, AI и базой

Форма должна быть описана декларативно.

Form config должен описывать:

- сущность;
- поля;
- типы полей;
- обязательность;
- допустимость AI-заполнения;
- требования к источнику;
- confidence на уровне поля;
- опасные поля;
- действия формы;
- какие действия требуют human click;
- какие действия пишутся в audit log.

Пример направления:

```ts
const personFormConfig = {
  entity: 'Person',
  title: 'Карточка человека',
  fields: [
    {
      key: 'fullName',
      label: 'ФИО',
      type: 'text',
      required: true,
      aiFill: true,
      sourceRequired: true,
    },
    {
      key: 'companyId',
      label: 'Компания',
      type: 'entity-select',
      entity: 'Company',
      aiFill: true,
    },
    {
      key: 'role',
      label: 'Должность / роль',
      type: 'text',
      aiFill: true,
      confidence: true,
    },
    {
      key: 'influenceMarker',
      label: 'Роль влияния',
      type: 'select',
      options: [
        'unknown',
        'lvr_candidate',
        'lpr_candidate',
        'lvr',
        'lpr',
      ],
      aiFill: true,
      humanConfirmRequired: true,
    },
    {
      key: 'source',
      label: 'Источник',
      type: 'source-picker',
      required: true,
    },
  ],
  actions: [
    {
      id: 'save_draft',
      label: 'Сохранить черновик',
      risk: 'low',
      humanClickRequired: true,
    },
    {
      id: 'confirm_lpr',
      label: 'Подтвердить ЛПР',
      risk: 'high',
      humanClickRequired: true,
      audit: true,
    },
  ],
}
```

Главная мысль:

```
Форма — это не только UI.
Форма — это контракт, по которому AI понимает,
что можно заполнить, что нужно подтвердить,
а что нельзя применять без человека.
```

## 14. Flow configs поверх form configs

Flow config описывает рабочий сценарий.
Form config описывает конкретный экран/форму.

Пример:

```ts
const addPersonFromVoiceFlow = {
  id: 'person.add_from_voice',
  trigger: 'voice_command',
  intent: 'add_person',
  steps: [
    {
      id: 'extract_person_data',
      actor: 'ai',
      output: 'PersonDraft',
    },
    {
      id: 'open_person_form',
      actor: 'system',
      form: 'PersonForm',
    },
    {
      id: 'prefill_form',
      actor: 'ai',
      target: 'PersonForm',
      mode: 'draft',
    },
    {
      id: 'human_review',
      actor: 'human',
      required: true,
    },
    {
      id: 'save',
      actor: 'human',
      action: 'entity.person.save',
    },
  ],
}
```

Важно:

```
AI может довести пользователя до готовой формы.
Но commit делает человек.
```

## 15. Граница ответственности AI

AI может:

- открыть форму;
- заполнить поля в черновике;
- предложить связь;
- найти похожие сущности;
- предложить merge;
- подготовить ping;
- подготовить встречу;
- подготовить сообщение;
- объяснить источник;
- показать последствия изменения;
- предложить следующий шаг.

AI не должен без человека:

- подтверждать ЛПР/ЛВР;
- делать merge;
- отправлять сообщение или приглашение;
- утверждать стратегию входа;
- утверждать отчет;
- менять confidence критичных фактов;
- закрывать конфликт источников;
- удалять или скрывать данные;
- применять изменения, которые меняют граф доверия.

Рабочая формула:

```
AI prepares.
Human commits.
System audits.
```

## 16. Поиск должен стартовать от любой сущности

Поиск не должен ограничиваться ИНН компании.

Система должна запускать расследование от разных стартовых точек:

| Стартовая точка | Пример вопроса |
|---|---|
| Company | Кто ЛПР и как зайти? |
| Person | Кого он знает и где может быть точкой входа? |
| Project / Object | Кто влияет на этот объект? |
| Tender / Lot | Через кого можно зайти в этот лот? |
| InternalUser | У кого из наших есть теплые контакты? |
| Meeting | Кого нужно позвать на созвон? |
| Document | Какие люди и связи из документа надо проверить? |
| Signal | Кого пинговать после изменения данных? |

Это превращает граф из “поиска по компании” в рабочую карту действий команды.

## 17. Командная координация: пинги и созвоны

Важная ценность может быть не в уникальном алгоритме,
а в ускорении ежедневной работы команды.

Система должна помогать:

- понять, кого из своих спросить;
- пингануть владельца контакта;
- подготовить короткий контекст для пинга;
- позвать нужных людей на созвон;
- подготовить agenda созвона;
- после созвона обновить сущности и связи;
- создать follow-up;
- напомнить, если ответа нет.

Пример flow:

```ts
const pingForIntroFlow = {
  id: 'team.ping_for_intro',
  goal: 'Получить интро к внешнему ЛПР через внутреннего владельца связи',
  inputs: ['TargetPerson', 'InternalOwner', 'Project'],
  steps: [
    'prepare_context_brief',
    'draft_ping_message',
    'human_approve_send',
    'track_response',
    'schedule_call_if_positive',
    'update_relationship_after_call',
  ],
  humanGates: [
    'send_message',
    'schedule_call',
    'update_relationship',
  ],
}
```

CRM-плагин может синхронизировать встречи и задачи наружу,
но логика “кого пингануть и зачем” должна быть понятна core-системе,
иначе AI не сможет вести процесс.

## 18. Chat-first workspace

Рабочая гипотеза по UX:

```
Правый чат — это не второстепенный помощник.
Правый чат — главный управляющий слой системы.
```

Пользователь общается с AI справа.
AI сам ведет пользователя по страницам, формам, графам и виджетам.
В перспективе AI может даже сам собирать временную рабочую поверхность
из нужных виджетов под конкретный flow.

Пример:

```
Пользователь: "делай"

AI:
1. открывает форму Person;
2. заполняет черновик;
3. открывает граф связей;
4. подсвечивает новую связь;
5. создает HITL-карточку;
6. просит человека подтвердить;
7. после подтверждения обновляет next steps.
```

То есть интерфейс становится не набором страниц,
а управляемым AI рабочим пространством:

```
Chat drives.
Widgets respond.
Human commits.
```

Для демо:

- не делать свободный полноценный AI;
- сделать 2 жестко заданных сценария;
- запускать их через заранее забитые команды;
- показывать, как AI “ходит” по уже существующим частям админки.

## 19. Demo flows: один про данные, один про коммуникацию

Для первого убедительного демо нужны два flow.

### Flow 1 — данные

Цель: показать, что AI помогает создавать и уточнять данные графа.

Пример:

```
делай: добавить человека как возможную точку входа
```

Смысл:

- AI открывает/заполняет форму Person;
- предлагает связь с Company/Project;
- создает или обновляет карточку в графе;
- создает HITL-решение на подтверждение ЛПР/ЛВР или связи;
- после решения человека данные реально меняются в mock-state.

### Flow 2 — коммуникация

Цель: показать, что AI ускоряет действия команды.

Пример:

```
делай: получить интро через внутреннего контакта
```

Смысл:

- AI анализирует существующую цепочку;
- готовит ping внутреннему владельцу связи;
- предлагает созвон;
- создает задачу/встречу;
- обновляет статус стратегии или next steps;
- человек подтверждает отправку/создание.

Данные после flow должны меняться по логике сценария.
Конкретные изменения можно уточнять итеративно.

## 20. Контексты human commit

Нравятся все три варианта подтверждения,
но применять их нужно по контексту.

### A. Подтверждение прямо в чате

Подходит для легких действий:

- создать черновик;
- открыть форму;
- продолжить flow;
- показать граф;
- подготовить текст.

### B. Подтверждение в форме

Подходит для entity/data changes:

- сохранить Person;
- сохранить Company;
- добавить Relationship;
- подтвердить источник;
- сохранить черновик с AI-prefill.

Для демо это обязательный паттерн:

```
AI заполняет форму → человек нажимает кнопку формы.
```

### C. Подтверждение через HITL-карточку

Подходит для решений, которые меняют доверие к графу:

- подтвердить ЛПР/ЛВР;
- принять связь как достоверную;
- разрешить конфликт источников;
- принять refresh signal;
- подтвердить merge;
- утвердить стратегию.

Для демо это тоже обязательный паттерн:

```
AI создает decision card → человек принимает решение.
```

Рабочее правило:

```
Chat confirmation = navigation / low-risk continuation.
Form confirmation = save structured data.
HITL card = trust-changing decision.
```

## 21. Демо-режим AI

Для ближайшей презентации не нужно притворяться,
что работает полноценный свободный AI.

Решение:

```
Демо-режим: две заранее забитые команды "делай".
```

Почему так лучше:

- сценарий контролируемый;
- директор видит ценность без случайных сбоев;
- можно показать будущую механику AI orchestration;
- не нужно реализовывать настоящий intent parser;
- существующие формы, графы, встречи и верификацию можно переиспользовать.

Формулировка в UI может быть честной:

```
Демо-сценарии
[делай: заполнить точку входа]
[делай: подготовить интро и созвон]
```
