# RIP — Proof of Concept
## Платный предварительный этап

---

## Цель PoC

Проверить две гипотезы на реальных данных заказчика **до** полноценной разработки MVP:

1. **Граф реально показывает пути?** Есть ли перекрёстные связи между проектами достаточные для shortest path?
2. **DaData даёт достаточно для обогащения?** ИНН есть у всех компаний? Или часть контактов — физлица и ИП без ИНН?

Если обе гипотезы подтверждены — идём в MVP. Если нет — знаем об этом через 2–3 недели, не через 3 месяца.

---

## Что делаем

### Шаг 1 — Ручной ввод данных
Менеджер вручную вносит 2–3 закрытых проекта:
- люди, компании, роли
- связи между ними
- теги ЛПР/ЛВР

Интерфейс: простая форма или даже AdminJS без кастомизации.
Объём: ~200 записей, 2–3 рабочих дня менеджера.

### Шаг 2 — Граф
PostgreSQL + recursive CTE. Никаких воркеров, никакого оркестратора.
Просто данные в таблицах и SQL запросы.

### Шаг 3 — DaData обогащение
Скрипт: берёт все Company, запрашивает DaData по ИНН, сохраняет результат.
Смотрим: сколько компаний нашлось, какое качество данных, есть ли руководители.

### Шаг 4 — Один реальный вопрос
```
"Вот новая компания X — есть ли путь до неё через известные контакты?"
```
Отвечаем запросом к графу. Показываем результат менеджеру.

### Шаг 5 — Визуализация
Простой HTML + vis.js. Никакого полноценного UI.
Менеджер видит граф своих реальных проектов на экране.

---

## Что НЕ делаем в PoC

| Компонент | Почему не делаем |
|-----------|-----------------|
| Keycloak | Не нужен для проверки гипотез |
| AdminJS кастомизация | Базового CRUD хватит |
| Telegram Bot | Не нужен |
| Refresh Worker / cron | Не нужен |
| Pseudonymization | Не прод, данные не уходят никуда |
| PDF Export | Не нужен |
| Чатбот | Не нужен |
| Orchestrator / job queue | Прямые вызовы, без очередей |
| AI Extraction | См. приписку в конце |

Human-in-the-loop в PoC — только разработчики. Менеджер только смотрит результат.

---

## Стек PoC

Минимальный, без архитектурных паттернов:

```
PostgreSQL — данные и граф
Node.js скрипт — DaData запросы
HTML + vis.js — визуализация
AdminJS — ввод данных
```

---

## Результат PoC

Показываем директору и менеджеру:
- Граф реальных проектов на экране
- Конкретный ответ: "через кого зайти в компанию X"
- Качество DaData обогащения — что система узнала о компаниях автоматически

После этого разговор простой:
> "Вот что система умеет на ваших реальных данных. Хотите полноценный MVP — вот стоимость и сроки."

---

## Как это защищает команду

PoC — отдельный контракт, фиксированная оплата, короткие сроки.

- Если после PoC заказчик говорит "не нужно" → потеряли 2–3 недели, не 3 месяца
- Если говорит "хотим дальше" → есть реальные данные, проверенный граф, понимание качества
- В любом случае работа оплачена

---

## Критерий успеха PoC

```
Менеджер смотрит на граф своих реальных проектов
и говорит: "да, это именно то что нам нужно"
```

Если этого не происходит — MVP делать не стоит.

---

## Переход к MVP

После успешного PoC:
- Данные уже в БД — не выбрасываем, используем как seed
- Граф проверен — архитектура Graph Layer подтверждена
- DaData коннектор написан — переезжает в MVP без изменений
- Понятен реальный объём данных и связей

## ⚠ Приписка про Extraction

AI extraction из PDF-отчётов намеренно исключён из PoC и вынесен как опциональный инструмент первичного онбординга в MVP.

**Почему не крит для старта:** ручной ввод 2–3 проектов — конечная работа, менеджер справится за 2–3 дня. После онбординга extraction в повседневной работе не нужен.

**Почему потенциально опасное место:**
- Качество LLM extraction на русскоязычных PDF с неформальными комментариями — неизвестно до проверки
- Если отчёты содержат сканы а не текстовый слой — extraction вообще не работает без OCR
- Ошибки extraction незаметны если нет верификации — граф тихо наполняется мусором
- Если в будущем захотят автоматизировать онбординг через extraction — это отдельная задача с отдельной оценкой

**Что нужно сделать перед реализацией extraction:**
Прогнать 2–3 реальных отчёта через выбранный LLM вручную. Оценить качество. Только потом принимать решение делать или нет.
