# Карта связей — Финальный MVP Роадмап
## Версия 1.1

---

## Главный архитектурный принцип — Connector Pattern

**Это главная долгосрочная ценность системы.**

Каждый внешний источник подключается через единый интерфейс:
```
interface SourceConnector {
  fetch(entityId, params): Observation[]
  capabilities(): string[]
  rateLimit(): RateLimitConfig
}
```

MVP: одна реализация — DaData.
V2: добавляем тендеры, новости, соцсети — каждый как новая реализация того же интерфейса.
Бизнес-логика не меняется. Граф обогащается автоматически.

**Для заказчика это звучит так:** "Система умнеет со временем без вашего участия."

---

**Project** — центральная сущность системы.

```
Project
├── подграф мастер-графа (люди и компании связанные с проектом)
├── Project Report (генерируется из подграфа)
└── метаданные (название, статус, даты, комментарии)
```

**Мастер-граф** — накопленная память о всех связях.
Каждый проект — это линза через которую смотришь на часть мастер-графа.

---

## Фаза 0 — Пре-запуск (до сдачи MVP)

**Кто делает:** разработчики, не менеджеры.
**Цель:** система не выходит к заказчику пустой.

### Шаги:
1. Берём 5–10 лучших закрытых проектов заказчика
2. Прогоняем PDF-отчёты через AI Extraction Worker
3. Вручную чистим результат
4. **Импортируем контакты из Outlook** — CSV экспорт → парсим → Person ноды с телефонами. Разовый скрипт, запускают разработчики.
5. Менеджер верифицирует через AdminJS (валидирует, не создаёт)
6. DaData автоматически обогащает все найденные компании по ИНН
7. Сдаём заказчику систему с уже заполненным графом

**Результат:** на момент демо директору — граф живой, данные реальные, система показывает ценность сразу.

**Важно:** это часть стоимости MVP, прописать в договоре отдельной строкой.

## Фаза 1 — MVP

### 1.1 Модель данных

#### Основные сущности (ноды графа)

| Сущность | Ключевые поля |
|----------|---------------|
| Project | id, name, status, created_at, signals_hash |
| Company | id, inn, canonical_name, aliases[], status |
| Person | id, canonical_id, full_name, aliases[] |

#### Связи (edges) — типы

| Тип | Описание | Уточнение |
|-----|----------|-----------|
| works_at | Person → Company | текущее место работы |
| worked_at | Person → Company | бывшее место работы |
| knows | Person ↔ Person | просто знакомы |
| colleagues | Person ↔ Person | работают в одной компании сейчас |
| ex_colleagues | Person ↔ Person | работали вместе раньше |
| partners | Person ↔ Person | взаимодействовали по работе/проектам |
| participates_in | Person/Company → Project | роль в проекте |
| affiliated_with | Company ↔ Company | структурная связь, партнёрство |

> ⚠️ Типы связей нужно уточнить с заказчиком до разработки.
> Их терминология может отличаться от нашей.

#### Теги на Person
- `is_lpr` — лицо принимающее решение
- `is_lvr` — лицо влияющее на решение
- `notes` — свободный текст менеджера

#### Data Pipeline (из старого ТЗ, адаптировано)

```
Источник
   ↓
Observation (сырые данные)
   ├── entity_id
   ├── source
   ├── raw_data
   ├── source_url    ← ссылка на источник обязательна
   └── observation_hash (UNIQUE per source+entity)
   ↓
Signal (событие)
   ├── "сменился директор"
   ├── "новый контракт"
   └── signal_key (UNIQUE per entity+type)
   ↓
Fact (структурный факт, версионируется SCD2)
   ├── valid_from
   └── valid_to
   ↓
Snapshot → Project Report
```

**Идемпотентность:**
- `UNIQUE(source_id, entity_id, observation_hash)` — нет дублей при повторном парсинге
- `signals_hash` на Project — не пересоздавать отчёт если граф не менялся

**Версионирование:** Temporal tables на всех сущностях (`valid_from` / `valid_to`). Ничего не удаляется.

---

### 1.2 Источники данных

#### MVP — один внешний коннектор: DaData

**Что даёт:**
- Полные данные компании по ИНН (название во всех вариантах, руководители, учредители, адрес, статус)
- До 10 000 запросов в сутки бесплатно
- Решает entity resolution для компаний через ИНН как anchor

**Когда вызывается:**
- При добавлении компании в проект (автоматически)
- При команде "обнови данные по компании X"
- Фоновый refresh по расписанию (раз в неделю)

**Ссылки на источники:** каждый Observation хранит `source_url`. В Project Report все факты имеют ссылку на источник.

---

### 1.3 AI слой

#### Extraction Worker
- Читает PDF-отчёты
- Извлекает: люди, компании, связи, роли, комментарии менеджеров
- Точность ~60–70% → остальное верифицирует менеджер

#### Pseudonymization Layer
```
Реальные данные → PERSON_001 / COMPANY_047 → LLM → ответ → обратная замена
```
LLM никогда не видит реальные имена, телефоны, ИНН.

#### LLM Connector
Абстракция провайдера. MVP: self-hosted (Ollama) или GigaChat.

#### Snapshot Worker
- Генерирует Project Report по шаблону
- Не пересоздаёт если `signals_hash` не изменился

---

### 1.4 Project Report — шаблон

```
1. Проект
   └── название, статус, даты

2. Ключевые участники
   └── люди и компании с ролями + ссылки на источники

3. ЛПР / ЛВР
   └── кто, через кого заходить, цепочка рукопожатий

4. Стратегии входа  ← новый раздел, генерирует AI
   └── Стратегия 1 (рекомендуемая):
         Путь: Менеджер → Иванова → Петров → Сидоров
         Риск: низкий
         Почему: друзья (5 лет) + бывшие коллеги + партнёры
         Источники: [ссылка на факт 1] [ссылка на факт 2]
   └── Стратегия 2 (рискованная):
         Путь: Менеджер → Смирнов → Сидоров
         Риск: высокий
         Почему: сотрудник к директору — неудобная позиция
         Источники: [ссылка на факт 1]
   └── Стратегия 3 (альтернативная): ...

5. Relationship Map
   └── визуальный рендер подграфа проекта

6. Сигналы и факты
   └── последние изменения (смена директора, новые контракты...)

7. Комментарии менеджера

8. Confidence summary
   └── насколько данные достоверны, когда обновлялись
```

AI заполняет все разделы по шаблону на основе метаданных графа.
Менеджер редактирует и утверждает. Экспорт в PDF.

**Как работает раздел "Стратегии входа":**
- Snapshot Worker берёт subgraph проекта
- Псевдонимизирует все PII
- Отправляет в LLM промпт с графом и метаданными (типы связей, веса, confidence, роли)
- LLM находит 2-3 стратегии и объясняет reasoning
- После депсевдонимизации — реальные имена и ссылки на source_url каждого факта
- Никакого нового модуля — это часть Snapshot Worker

**Условие активации:**
Раздел генерируется только если граф проекта содержит минимум одну ноду с тегом `is_lpr` или `is_lvr` и минимум 3 edges с confidence > 0.5. Иначе — заглушка "недостаточно данных для анализа стратегии".

### 1.5 Чатбот — 8 команд

Чатбот работает через intent matching. Никакого свободного диалога в MVP.
Каждая команда транслируется в конкретный запрос к БД/графу.

| # | Команда | Что происходит |
|---|---------|----------------|
| 1 | "Создай проект [название]" | INSERT в projects, ответ с подтверждением |
| 2 | "Добавь компанию [ИНН/название] в проект X" | DaData запрос → нода создана/найдена → связь с проектом |
| 3 | "Найди точку входа для компании X" | Shortest path + weighted path в графе |
| 4 | "Как зайти в компанию X?" | Snapshot Worker → раздел "Стратегии входа" → структурированный ответ с reasoning и ссылками |
| 5 | "Кто ЛПР/ЛВР в проекте/компании X" | Выборка нод с тегами is_lpr / is_lvr |
| 6 | "Покажи связи по проекту X" | Подграф проекта |
| 7 | "Покажи/сгенерируй отчёт по проекту X" | Если signals_hash актуален → PDF. Если нет → пересоздать → PDF |
| 8 | "Обнови данные по компании X" | DaData refresh → новые Observations → Signal/Fact update |

### 1.6 Feedback (минимальный)

Менеджер может отметить данные: 👍 полезно / 👎 не релевантно / ⚠ сомнительно.

В MVP: только запись в `feedback_events`. Автоматически не применяется.
В v2: feedback влияет на source_weight и ранжирование сигналов.

### 1.8 Refresh Worker (минимальный silent worker)

Единственный фоновый процесс в MVP. Решает проблему adoption — система сама приходит к менеджеру, не наоборот.

**Логика:**
```
Каждую ночь (cron):
→ берёт все компании у которых есть активные проекты
→ DaData запрос по каждой
→ сравнивает с последним Observation (через hash)
→ если изменилось → создаёт Signal
→ если Signal создан → Telegram уведомление менеджеру
```

**Что отслеживает (через DaData):**
- Смена директора
- Смена учредителей
- Изменение статуса компании (ликвидация, реорганизация)
- Смена адреса

**Уведомление в Telegram:**
```
⚡ Сигнал по проекту "Альфа"
Компания: ООО Стройгрупп
Событие: сменился генеральный директор
Было: Иванов И.И. → Стало: Петров П.П.
Источник: ЕГРЮЛ / DaData [ссылка]
```

**Почему это важно для MVP:**
Меняет паттерн с pull ("я иду в систему когда нужно") на push ("система сама говорит когда важное"). Без этого система остаётся отдельным инструментом который надо помнить открывать.

### 1.9 Инфраструктура

| Компонент | Решение |
|-----------|---------|
| БД | PostgreSQL (граф через recursive CTE) |
| Очереди | PostgreSQL jobs (MVP) → Redis + BullMQ (v2) |
| Auth | Keycloak |
| LLM | Ollama self-hosted или GigaChat |
| Внешний коннектор | DaData API |
| Версионирование | Temporal tables (valid_from / valid_to) |
| Уведомления | Telegram Bot API |
| Фоновые задачи | Cron (Refresh Worker) |

## V1 — умная аналитика

После MVP, отдельный этап.

- Стратегии входа в отчёте — AI анализирует граф, даёт 2-3 варианта с reasoning и ссылками на источники
- Relationship Map в отчёте — визуальный рендер подграфа
- Weighted path визуализация — показывает оба пути (кратчайший и качественный)
- Feedback система — менеджер оценивает данные (👍/👎/⚠), сразу меняет confidence
- Conflict resolution UI — очередь конфликтов когда два источника дают разные данные

## Критерий приёмки MVP

Один сценарий от начала до конца:

```
Менеджер: "Создай проект Альфа"
Менеджер: "Добавь компанию 7730588444 в проект Альфа"
  → система: нашла ООО Стройгрупп, добавила, обогатила через DaData
Менеджер: "Найди точку входа для компании Стройгрупп"
  → система: PERSON_003 знает PERSON_007, который участвовал в проекте 2021
Менеджер: "Покажи отчёт по проекту Альфа"
  → система: генерирует PDF с картой связей и точками входа
```

Если этот сценарий работает на реальных данных заказчика — MVP принят.
